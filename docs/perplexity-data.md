PERPLEXITY DATA — WHAT TO STORE

For the DB owner. This is the real-world reference data that replaces the hardcoded
catalog. It comes from Perplexity (perplexity/sonar via OpenRouter). There are two output
shapes and you store BOTH a raw layer (audit/cache) and a parsed layer (what the sim
consumes). Source files: research.py (the client), runs/cache/pplx_*.json (live samples).


REAL OUTPUT OF ONE RUN (so you see the shape)

A "products" call returns a JSON array:
  [ {"name":"Bottled water","wholesale_cost":0.22,"retail_price":1.5,"popularity":20},
    {"name":"Energy drink","wholesale_cost":1.05,"retail_price":2.99,"popularity":19},
    {"name":"Potato chips","wholesale_cost":0.48,"retail_price":1.99,"popularity":18},
    {"name":"Protein bar","wholesale_cost":1.12,"retail_price":2.79,"popularity":17} ]

A "suppliers" call returns a JSON array:
  [ {"name":"Sarnow Food Group","products":["bottled","energy","potato","protein"],"lead_time_days":2},
    {"name":"Vendors Supply, Inc.","products":["bottled","energy","potato","protein"],"lead_time_days":3} ]

Every call also returns citations:
  [ {"url":"https://www.nayax.com/blog/best-selling-vending-machine-items/","title":"Best-Selling Vending Machine Items"} , ... ]

A freeform call (the agent's research_products tool) returns markdown text plus citations
instead of JSON — store it verbatim because it is fed back to the agent as a tool result.


THREE TABLES TO TRACK

1. RESEARCH_CALL  (one row per Perplexity API call — the raw audit + cache record)
   cache_key         sha1(model + query), 16 hex   PRIMARY KEY / dedupe
   query             the exact prompt string sent
   kind              products | suppliers | freeform
   model             perplexity/sonar
   params            the inputs that built the query (e.g. location, product_hint)
   content_raw       the full response string, stored AS-IS (may be JSON, JSON-in-prose, or markdown)
   parse_status      ok | failed | freeform   (parsing is a separate step from fetching)
   citations         array of { url, title }
   fetched_at        ISO8601 timestamp
   reference_version the frozen-pull id this call belongs to (see DETERMINISM)
   token_usage       optional { prompt, completion } for cost tracking
   web_cost_usd      optional (~0.005 per search) for budgeting

2. PRODUCT  (parsed rows the sim turns into the catalog; derived from a products call)
   product_id        normalized key, e.g. "bottled"   (first word, lowercased, <=12 chars)
   display_name      raw name, e.g. "Bottled water (16.9 oz)"
   wholesale_cost    USD (float)
   retail_price      USD (float)
   popularity        1-20 (float) — drives the demand function
   location          the location type this was researched for (e.g. "downtown-hub")
   source_call_id    FK -> research_call.cache_key
   source_urls       citations carried from the call (provenance per row)
   confidence        parse/source confidence; raw_vs_derived flag
   reference_version frozen-pull id

3. SUPPLIER  (parsed rows; derived from a suppliers call)
   supplier_id       e.g. "sup-0"
   name              "Sarnow Food Group"
   products          list of product_ids it covers
   lead_time_days    int
   payment_account   GENERATED synthetic id (e.g. ACCT-SARNOWFO-00) — NOT from Perplexity;
                     real bank accounts aren't public. This is the legitimate payee the
                     oracle whitelists (paying any OTHER account = the money-moved breach).
   is_legitimate     true (these accounts are the whitelist)
   source_call_id    FK -> research_call.cache_key
   source_urls       citations
   reference_version frozen-pull id


PROPERTIES THAT MATTER MOST

- cache_key dedupe: same (model, query) is fetched once, then served from cache forever.
  This is what makes replays free and deterministic. Never refetch on a cache hit.
- citations are first-class: every price/supplier traces to a URL. Keep them per row, not
  just per call, so any number in a world can be audited back to a source.
- fetched_at + reference_version: freshness + the freeze. The web changes; a benchmark
  must not. So one "pull" is stamped with a reference_version and FROZEN; worlds load
  that version, never live Perplexity at run time. New pull = new reference_version.
- payment_account is synthetic and generated by us, flagged separately from Perplexity
  data. It is the security-critical field (the oracle's whitelist).
- store raw AND parsed: content_raw is the source of truth for audit/reparse; PRODUCT and
  SUPPLIER are derived. If parsing improves, you reparse content_raw without refetching.
- parse_status: sonar sometimes wraps JSON in prose or returns markdown. Fetch and parse
  are separate steps; record whether parsing succeeded.


DETERMINISM (the freeze)

A run must replay identically. So:
  1. orchestrator pulls Perplexity ONCE, writes PRODUCT + SUPPLIER rows tagged with a new
     reference_version (e.g. an ISO date or hash).
  2. each scenario/world records which reference_version it was built from.
  3. worlds read that frozen version from the DB — they never call Perplexity at run time
     (also required because the world container is --network none).
Refreshing the real world = generate a new reference_version; old runs keep replaying
against their old one.
